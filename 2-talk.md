## 🎯 核心協作原則

### 1. 理解優先，代碼其次
- ✅ **永遠先理解問題本質**，再開始寫代碼
- ✅ 如果不確定需求，**必須先詢問**，不要猜測
- ✅ 複雜問題先**畫流程圖**或**寫偽代碼**
- ❌ 不要急於動手，避免返工

### 2. 通用解決方案
- ✅ 寫**可擴展**、**可維護**的代碼
- ✅ 考慮**邊界情況**和**錯誤處理**
- ✅ 遵循 **DRY 原則**（Don't Repeat Yourself）
- ❌ 不要硬編碼，不要寫一次性代碼

### 3. 增量開發
- ✅ 把大任務**拆分成小步驟**
- ✅ 每個步驟都要**可測試**
- ✅ 及時**反饋進度**和**遇到的問題**
- ❌ 不要一次性完成所有功能

### 4. 代碼質量
- ✅ 代碼要**清晰易讀**，勝過聰明技巧
- ✅ 保持函數簡短（**<50 行**）
- ✅ 使用**有意義的命名**
- ✅ 添加**必要的註釋**（解釋"為什麼"）

## 🔧 代碼規範

### 命名規範

**文件命名**：
- 使用項目慣例（snake_case 或 kebab-case）
- 測試文件：`test_*.ext` 或 `*.test.ext`

**變量命名**：
- JavaScript/TypeScript/Dart：`camelCase`
- Python：`snake_case`
- 常量：`UPPER_SNAKE_CASE` 或 `camelCase`（根據語言）

**類命名**：
- 所有語言統一：`PascalCase`

**函數命名**：
- 動詞開頭：`getUserData()`, `calculateTotal()`, `isValid()`
- 布爾值：`is*`, `has*`, `should*`

### 註釋規範

**必須註釋的地方**：
```javascript

// 根據會員等級計算折扣：VIP 9折，普通會員 95折
const discount = user.isVIP ? 0.9 : 0.95;

// ✅ 臨時的變通方案
// TODO: 等 API v2 發布後改用新的端點
// FIXME: 這個方法在 IE11 中有問題
// HACK: 暫時用這個方法繞過 CORS 錯誤
```

**不需要註釋的地方**：
```javascript

// 設置用戶名為 John
const userName = 'John';

// ❌ 復述代碼內容
// 循環遍歷數組
```

### 錯誤處理

**必須處理錯誤的場景**：
```javascript
// ✅ 網絡請求
try {
  const data = await fetch(url);
  return data;
} catch (error) {
  console.error('❌ 請求失敗:', error);
  // 顯示用戶友好的錯誤訊息

// ✅ 文件操作
try {
  const content = await readFile(path);
  return content;
} catch (error) {
  if (error.code === 'ENOENT') {
    console.warn('⚠️ 文件不存在:', path);
    return createDefaultFile(path);
  }
  throw error;
}

// ✅ 用戶輸入
if (!email || !email.includes('@')) {
  throw new Error('請輸入有效的電子郵件地址');
}
```

```javascript
// ✅ 清晰的測試描述
test('calculateDiscount 應該在用戶是 VIP 時返回 10% 折扣', () => {
  expect(calculateDiscount({ isVIP: true, total: 100 })).toBe(90);

test('calculateDiscount 應該處理 0 金額', () => {

});
```

```javascript
 * 計算用戶折扣
 * 
 * @param {Object} user - 用戶對象
 * @param {boolean} user.isVIP - 是否為 VIP
 * @param {number} total - 總金額
 * @returns {number} 折扣後的金額
 * 
 * @example
 * const finalPrice = calculateDiscount({ isVIP: true }, 100);
```

當前的廣告邏輯主要分為 插頁式廣告 (Interstitial) 與 獎勵式廣告 (Rewarded) 兩部分，分別針對「AI 分析流程」與「獲取額度」設計。

以下是詳細的技術邏輯拆解：

1. 核心服務 (
AdService
)
單例模式：由 
AdService
 統一管理廣告載入與顯示。
ID 配置：目前 Android 使用 Google 測試 ID，iOS 使用生產環境 ID (AdConfig.dart)。
頻率控制：代碼中設定了 _interstitialFrequency = 1，意味著只要符合條件，每次觸發都會嘗試顯示廣告。
2. 插頁式廣告 (Interstitial) - AI 分析前
這是在用戶進行食物拍照/上傳分析時觸發的強制性廣告。

觸發點：MainNavigationScreen.dart 中的 
_analyzeImage
 方法。
觸發條件：
用戶是 免費會員 (MembershipTier.free)。
AdService
 的頻率檢查通過 (shouldShowInterstitial)。
平行加載優化 (Parallel Loading)：
為了減少等待感，系統會同時發送「Gemini AI 分析請求」與「廣告載入請求」。
延遲策略：為了 UX 體驗，強制等待至少 0.5 秒轉圈圈動畫，避免廣告瞬間閃現過於突兀。
顯示流程：
如果廣告在分析完成前載入成功 -> 顯示廣告 -> 用戶關閉廣告 -> 顯示分析結果。
如果廣告載入失敗或太慢完成 -> 直接顯示分析結果 (不阻擋用戶)。
3. 獎勵式廣告 (Rewarded) - 獲取積分
這是在用戶主動點擊時觸發的廣告，用於獲取額外的 AI 使用次數。

觸發點：MembershipScreen.dart 中的「觀看廣告賺取積分」區塊。
功能邏輯：
限制：僅免費會員可見。
機制：點擊「立即觀看」-> 載入並播放影片廣告。
獎勵：觀看完成後，調用 MembershipProvider.addRewardPoints(1) 增加 1 點 purchasedCredits。
防刷機制：
目前的邏輯 (
addRewardPoints
) 似乎設定了上限：如果你的剩餘額度 (remainingAiQuota) 已經達到或超過每月上限 (10次)，則無法繼續賺取積分。這意味著它主要用於**「補滿」**免費額度，而不是無限累積。
4. 會員判斷邏輯 (
MembershipProvider
)
showAds 屬性：只有當 _tier == MembershipTier.free 且沒有被調試模式強制關閉時，才為 true。Premium 用戶完全不會觸發廣告加載邏輯。
總結來說： 現在的邏輯是 「免費與優化並行」：免費版用戶在每次分析等待期間會看到插頁廣告（利用分析的等待時間載入），並且可以透過主動看廣告來補充電量（額度）。Premium 用戶則完全免除這些干擾。
